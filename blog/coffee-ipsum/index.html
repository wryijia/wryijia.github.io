1.2 产品简介

1.2.1产品类型：

互联网校园信息共享平台

1.2.2产品logo:

1.2.3 产品介绍：

平台针对庞杂的校园信息进行汇总分类，用户通过小程序，根据需求选择服务类型，是为学生，教职工，学校和校周边用户群体提供信息便利的一个平台。

1.2.4 产品定位：

以校园为中心实现生活、学习和交友的信息共享服务平台，包括“我要表白”、“失物招领”、“闲置出让”、“树懒树洞”、“组团”、“资料下载”等各类信息，通过线上信息平台，实现信息的高效利用和流通的服务系统。

1.2.5 产品背景：

在大学校园生活里，极大量的需求信息喷涌而出，然而，校内信息共享平台众多，消息冗杂，寻找需求信息需耗费一定的心力，信息需求者与供应者总是失之交臂，彼此寻找，不能有效实现信息的流通与对接。目的就是将这些庞杂的信息汇总起来，进行分类，优化平台服务，从解决校园生活的各种需求出发，使信息能够高效的流通起来，真正为信息用户服务到位。

1.3 产品用户

学生，教职工，学校和校周边用户群体

1.4 需求总结

根据KANO模型，将用户需求分为五类：基本型需求、期望型需求、兴奋型需求、无差异型需求以及反向型需求。

五、 运营方案

5.1 对于未来的战略构想

内容为王→网络外联→矩阵平台→裂变增长

5.1.1 内容为王

  • 实用、有效、符合大众的需求是我们目前所需要去强化的一个地方，我们要主动学习和阅读；也要去多观察和思考。
  • 内容输出
  • 学习运营

5.1.2 网络外联

5.1.2.1 价值基础
  • 挖掘学生刚性需求
5.1.2.2 保持互动

  • 拉新和促活。

  • 吸引到人，能产生共鸣。

5.1.2.3 媒体矩阵
  • 分为纵向矩阵和横向矩阵；纵向矩阵是联系学院、各系、各组织以及有影响力的账号进行一对一、一对多、多对一的互推。进行流量增长；横向推广则是通过QQ、贴吧、微博等平台进行联动宣传，曝光品牌来更多的吸引关注群体。
5.1.2.4 分析流量
  • 在于所关注的群体，通过用户画像（就是分析主要关注的对象、使用的手机、男女比例、分布地区等等）。
5.1.2.5 媒体矩阵
  • 矩阵平台最高的发展形势是高校联盟，高校联盟中的会员进行相互竞争和比拼可以带动地区中校园新媒体的比重和分量，纵观发达地区的大学已经具备这样的基础和条件，如何将资源和外部的资源进行整合和利用就是我们发展中需要考虑的一个问题。
5.1.2.6 网络外联
  • 就是通过基本条件的构成，形成大学生的一个信息取得群体，联合商家做出大学生的优惠活动，为商家带来品牌曝光和经济利益的基础上来赚取学生活动的经费，用于学生活动、媒体宣传、运营维护等费用。

4、裂变增长

  • 通过H5、快闪视频、短视频、主页设计等方式手段进行吸引流量和粉丝，这只是手段。核心是策划可行性的裂变活动，用什么去吸引用户、用什么去达到效果是我们预想的东西。

  • 当然，活动是一方面，另一方面就是社群运营，社群运营简而言之就是把关注的用户聚集在一起建立规则和制度，讨论或交流一个经常性的话题，比如说明星类、游戏类等。

6.1 客户端

6.1.1 小程序端

使用weUI框架

6.1.2 服务端

1. 云函数

一段运行在云端的代码，无需管理服务器，在开发工具内编写、一键上传部署即可运行后端代码。 小程序内提供了专门用于云函数调用的 API。开发者可以在云函数内获取到每次调用的上下文（appid、openid 等），无需维护复杂的鉴权机制，即可获取天然可信任的用户登录态（openid）。 开放了运用 Node.js 等框架编写“后端”业务逻辑后，直接可以部署在云平台下，完全不需要去考虑域名、服务器、打包发布、运维等等琐事

2. 云数据库

云开发提供了一个 JSON 数据库，顾名思义，数据库中的每条记录都是一个 JSON 格式的对象。一个数据库可以有多个集合（相当于关系型数据中的表），集合可看做一个 JSON 数组，数组中的每个对象就是一条记录，记录的格式是 JSON 对象。 这样的话，数据库的存储也不用考虑了，直接提供了一个类似于 MongoDB一样的数据库

3. 云存储

云开发提供了一块文件存储空间，提供了上传文件到云端、带权限管理的云端下载能力，开发者可以在小程序端和云函数端通过 API 使用云文件存储功能。 在小程序端可以分别调用 wx.cloud.uploadFile 完成上传 这就相当于腾讯直接给你提供了一个静态文件的 CDN

4. 云调用

云调用是云开发提供的基于云函数使用小程序开放接口的能力，支持在云函数调用服务端开放接口，如发送模板消息、获取小程序码等操作都可以在云函数中完成
